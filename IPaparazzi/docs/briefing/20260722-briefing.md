# Project Briefing: IPaparazzi

**Data**: 2026-07-22
**Status**: Aprovado
**Versão**: 1.0

---

## 1. Visão e Propósito

**O que é**: monitor multiplataforma do IPv4 público da conexão, capaz de manter registros DNS sincronizados quando esse endereço mudar.

**Problema que resolve**: conexões com IPv4 dinâmico podem deixar serviços inacessíveis quando registros DNS continuam apontando para um endereço antigo, especialmente quando a máquina está atrás de firewall, roteador ou outro equipamento de borda.

**Proposta de valor**: automatizar a descoberta confiável do IPv4 externo e corrigir somente os registros afetados, com baixo volume de chamadas aos provedores e rastreabilidade operacional.

## 4. Escopo

### MVP (Essencial)

1. Consultar três fontes fixas e independentes para descobrir o IPv4 público: Cloudflare, AWS e ipify.
2. Aceitar o IPv4 somente quando pelo menos duas fontes retornarem exatamente o mesmo endereço público válido.
3. Rejeitar endereços inválidos, privados, reservados, loopback ou multicast e não alterar DNS sem consenso.
4. Suportar uma lista de registros `A`, organizada por provedor e por conta, inclusive várias contas do mesmo provedor.
5. Integrar inicialmente apenas com a Cloudflare, preservando uma organização que permita adicionar outros provedores posteriormente.
6. Atualizar somente registros existentes; registro ausente deve falhar com segurança sem ser criado automaticamente.
7. Configurar o estado `proxied` individualmente para cada registro da Cloudflare.
8. Isolar falhas por registro ou subdomínio e continuar processando os demais.
9. Persistir localmente o último IPv4 aplicado e a última confirmação de cada registro.
10. Evitar consultas à API DNS quando o IPv4 não mudou e a última confirmação tiver menos de 24 horas.
11. Após 24 horas, reconciliar o estado local com o provedor para detectar alterações externas.
12. Registrar cada execução, o IPv4 detectado e o resultado de cada registro: confirmado, atualizado, ignorado ou com erro.
13. Rotacionar os logs automaticamente, por padrão mantendo cinco arquivos de até 5 MB cada, com valores configuráveis.
14. Integrar com o NotiCLI para notificar mudança de IP, falha parcial ou total, ausência de consenso e recuperação após falha.
15. Fornecer instaladores idempotentes para `cron` no Linux e Agendador de Tarefas no Windows, com intervalo informado pelo usuário.
16. Fornecer arquivo `IPaparazzi.toml.model` versionado e usar `IPaparazzi.toml`, ignorado pelo Git, para a configuração real e os tokens.

### Pós-MVP (Desejável)

1. Adicionar outros provedores DNS.
2. Adicionar suporte a IPv6 e registros `AAAA`.
3. Avaliar fontes adicionais de descoberta de IP caso as três fontes iniciais deixem de oferecer disponibilidade suficiente.

### Fora de Escopo

- Criar automaticamente registros DNS ausentes.
- Atualizar registros `CNAME` como se contivessem endereços IP.
- Usar Cloudflare Spectrum para serviços TCP/UDP não atendidos pelo proxy DNS convencional.
- Instalar `systemd timer` no MVP.
- Disponibilizar interface gráfica, web ou móvel.

## 5. Prioridades e Trade-offs

**Ordem de prioridade**: segurança operacional e robustez > baixo volume de chamadas externas > portabilidade > velocidade de entrega `[inferido]`.

**Decisões explícitas**:

- Exigir maioria de duas entre três fontes, aceitando deixar de atualizar quando não houver consenso para evitar publicar endereço incorreto.
- Falhar apenas o registro afetado para que um problema localizado não impeça as demais atualizações.
- Não consultar o provedor em toda execução quando o estado recente já comprovar que o DNS está correto.
- Usar somente `cron` no Linux porque a tarefa é simples e não exige os recursos adicionais de um `systemd timer`.
- Manter o token no TOML real ignorado, seguindo o padrão existente do repositório, em vez de obrigar variáveis de ambiente.

## 8. Qualidade e Padrões

**Padrões adotados**:

- Validar configuração antes de executar chamadas externas.
- Aplicar timeouts a toda chamada HTTP e registrar falhas sem expor tokens.
- Atualizar somente após validar IPv4 público e obter o consenso mínimo.
- Preservar o processamento dos demais registros diante de falha localizada.
- Gravar estado de maneira resistente a interrupções; o mecanismo exato será definido no planejamento.
- Produzir logs rotativos e notificações somente para eventos relevantes, evitando ruído em execuções normais.
- Manter instaladores idempotentes para não duplicar agendamentos.
- Cobrir fluxos válidos, entradas inválidas, limites, falhas externas, consenso, cache de 24 horas e atualizações parciais com testes unitários.
- Documentar objetivo, configuração, segurança, instalação, uso, limitações e diagnóstico no README interno.

**Compliance**: nenhum requisito regulatório específico informado.

## 9. Visão de Futuro

**6 meses**: núcleo estável em Windows e Linux, operando várias contas e registros Cloudflare com observabilidade suficiente para uso recorrente `[inferido]`.

**12 meses**: inclusão de novos provedores DNS e possível suporte a IPv6, preservando o mesmo contrato de configuração e execução `[inferido]`.

**Riscos conhecidos**:

- Indisponibilidade ou mudança de contrato nas fontes externas de descoberta do IP.
- Divergência temporária entre fontes impedir uma atualização legítima.
- Tokens com permissões excessivas ou arquivo TOML real com permissões inadequadas.
- Alteração manual do DNS durante a janela de até 24 horas permanecer temporariamente sem reconciliação.
- Registros com `proxied = true` não atenderem protocolos ou portas incompatíveis com o proxy convencional da Cloudflare.
- Execuções concorrentes corromperem ou sobrescreverem o estado local se não houver exclusão mútua.
